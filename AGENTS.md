# Repository Working Rules

## Before starting a task

1. Read `README.md`, the relevant experiment plan or validation document under `docs/`, and the active configuration under `configs/`.
2. Run `git status --short` before modifying files, and preserve unrelated user changes.
3. Do not develop new features directly on `main`. Use one clearly named feature branch per task.

## Git safety

1. Do not run `git push` unless the user explicitly requests it.
2. Do not rewrite commit history unless the user explicitly requests and approves the exact operation.
3. Never run `git reset --hard`, `git clean -fd`, or a force push.
4. Do not commit datasets, model weights, generated outputs, logs, caches, virtual environments, IDE state, or credentials.
5. Review `git status --short`, `git diff --stat`, and a dry-run staging preview before each commit.

## Research integrity

1. Experiment results must be generated by the checked-in script and configuration; never fabricate outputs, metrics, logs, or test results.
2. Keep experiment scope explicit. Do not present scene generation as prediction, planning, control, or training work.
3. Keep seeds, coordinate conventions, data-splitting limitations, and known sources of leakage documented.
4. Never place secrets, tokens, passwords, private keys, or credential-bearing URLs in source files or configuration.

## Validation and reporting

1. After a change, run the relevant unit tests and smoke experiment using the documented commands.
2. Report the actual commands, exit status, test results, `git diff --stat`, and unresolved issues.
3. If validation fails, report the failure honestly, fix only the in-scope problem, add a regression test when appropriate, and rerun validation.
4. Do not claim completion while required checks are failing or while results have not actually been generated.
